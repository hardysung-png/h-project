## 1. 개요

이 문서는 `prd.md`·`design-doc.md`가 다루지 않는 V0.1 이후의 진화 경로를 버전 단위로 정리한 로드맵이다.

### 로드맵 원칙

1. **핵심 가치는 V0.1에서 완결한다.** 차등 N빵 + 일정별 응답이 V0.1의 완결 형태다. 이후 버전은 그 경험을 더 안전하게, 더 널리, 더 오래 사용되게 만드는 방향으로만 간다.
2. **개인 프로젝트 리듬을 유지한다.** 큰 외부 API 연동·다국어는 뒤로 민다. 한 버전당 범위를 작게 잡는다.
3. **각 버전은 한 문장 주제로 설명 가능해야 한다.** 주제가 한 문장에 안 잡히면 범위가 지나치게 넓다.

## 3. V0.1 — 핵심 가치 검증 (CURRENT)

> 정본: `prd.md`. 아래는 로드맵 맥락을 위한 요약이다.

**주제**: 한 이벤트 + 일정별 참석 응답 + 차등 N빵 정산을 완성한다.

### 포함 기능

| 기능 | 설명 |
|------|------|
| 이벤트 생성 | 제목 + 세부 일정 최대 3개(이름 + 날짜/시간) |
| 초대 링크 | 이벤트마다 고유 URL 자동 생성, 복사 버튼 제공 |
| 참석 응답 | 닉네임 + 어른/아이 선택 + 일정별 참석/불참 (회원가입 불필요) |
| 응답 수정 | 같은 링크에서 같은 닉네임으로 재제출 → 덮어쓰기 |
| 참석 집계 | 호스트 대시보드에서 일정별 참석 인원 테이블로 확인 |
| 비용 입력 | 항목명 + 금액 (여러 개 추가 가능) |
| 차등 N빵 정산 | 어른 1인분, 아이 0.5인분, 특정인 제외 — 자동 계산 |
| 정산 제외 설정 | 참석자별로 정산 제외/포함 토글 |

### V0.1 완료 기준

- [ ] 호스트 회원가입/로그인
- [ ] 이벤트 "9월 가족모임" + 일정 2개(점심 모임, 차담회) 생성
- [ ] 초대 링크 복사 동작
- [ ] 참석자 4명 각각 다른 조합으로 응답 (엄마·아들·막내조카·삼촌)
- [ ] 호스트 대시보드 "점심 3/4, 차담 3/4" 정확 표시
- [ ] 비용 입력: 식비 150,000 + 장소비 50,000 + 케이크 30,000
- [ ] 호스트 정산 제외 설정
- [ ] 차등 N빵 결과 수치 정확 (3.5인분 기준)
- [ ] Vercel 배포 후 동일 시나리오 재통과

## 4. V0.2 — 신뢰성과 실시간

**주제**: 응답이 조작되지 않는다. 참석 현황이 즉시 반영된다.

V0.1은 "닉네임 동일 = 덮어쓰기" 규칙으로 동작한다. 이 구조는 간단하지만 악의적 덮어쓰기에 취약하고, 변경 사항을 확인하려면 새로고침이 필요하다. V0.2는 이 두 가지를 해소한다.

### 포함 기능

| 기능 | 설명 |
|------|------|
| 참석자별 signed token | 최초 응답 시 개인 토큰 발급. 이후 수정은 토큰 검증 필요. 닉네임 탈취 방지. |
| Supabase Realtime | 참석 현황 실시간 반영. 호스트 대시보드 새로고침 불필요. |
| 일정별 정산 | 이벤트 전체 단일 정산 → 일정별 참석자 기준 분할 정산. |
| 호스트 정산 제외 기본값 개선 | 호스트를 기본 "제외"로 설정하는 UX 옵션 추가. |

> PRD 9섹션 "미해결 질문" 1번(응답 수정 보안), 3번(일정별 정산), 4번(실시간 반영)이 여기서 해소된다.

## 5. V1.0 — 초대 경험 확장

**주제**: 단톡방 링크 한 줄의 한계를 넘는다.

V0.2까지는 카카오톡 링크 복사 + 모바일 브라우저 접속만 지원한다. V1.0은 초대 채널과 호스트 권한을 확장해 가족 밖(친척·지인) 이벤트까지 쓸 수 있게 만든다.

### 포함 기능

| 기능 | 설명 |
|------|------|
| 초대장 테마 디자인 | 생일 / 팔순 / 가족모임 / 일반 테마. 링크 클릭 첫 인상을 개선한다. |
| 이메일 초대 | Resend 연동. 이메일 주소만 입력하면 초대장 발송. |
| 카카오 알림톡 | 카카오톡 채널 연동. 알림톡 발송. |
| 부호스트 권한 | 이벤트 관리 권한을 다른 호스트와 공유. |

> 외부 API 의존 항목(카카오 알림톡)은 정책 변경 리스크가 있어 V0.x로 당기지 않는다.

---

## 6. V2.0 — 관계 자산화

**주제**: 이벤트가 쌓이면 지인 네트워크의 기록이 된다.

V1.0까지는 각 이벤트가 독립적으로 존재한다. V2.0은 반복 등장하는 참석자를 추적해 "함께한 사람들의 역사"를 쌓는다. 이벤트 누적이 충분할 때만 의미 있는 기능이므로 이 시점으로 미룬다.

### 포함 기능

| 기능 | 설명 |
|------|------|
| 지인 목록 | 이벤트에 반복 등장하는 사람 자동 집계. 닉네임 기반 매핑. |
| 함께한 모임 로그 | 지인별 참석 이벤트 타임라인. |
| 반복 이벤트 | 매년 명절·생일처럼 정기적으로 열리는 이벤트 템플릿. |
| 갤러리 / 사진 / 후기 | 이벤트 사진 첨부 및 짧은 메모 남기기. |

## 7. V3+ — 확장

| 후보 기능 | 방향 |
|-----------|------|
| 다국어 지원 | 한국어 전용 → 해외 거주 가족·교포 커뮤니티까지. |
| 앱 푸시 알림 | 참석 응답 마감 알림, 비용 변경 알림. |
| 정산 결제 연동 | 외부 결제 서비스와 연동해 실제 송금까지 처리. |
| 정산 리마인드 | 미정산 참석자에게 자동 알림. |

V3+는 현재 미확정 후보다. V2.0 이후 제품 방향에 따라 우선순위가 바뀔 수 있다.

---

## 8. 우선순위 판단 근거

**왜 V0.2 → V1.0 → V2.0 → V3+ 순서인가.**

1. **신뢰성 우선 (V0.2)**: 가족 모임에서 "돈 계산"이 잘못되면 한 번의 실수로 신뢰를 잃는다. 닉네임 덮어쓰기 보안 구멍은 초대 채널 확장보다 먼저 닫아야 한다.

2. **경로 확장 다음 (V1.0)**: 신뢰성이 확보된 뒤에야 단톡방 밖 초대 채널을 늘리는 게 의미 있다. 반대 순서라면 확장된 채널에서 온 사람이 조작에 노출된다.

3. **관계 자산화 마지막 (V2.0)**: 지인 목록·타임라인은 이벤트가 최소 수십 개 쌓여야 제공 가능한 기능이다. 이른 시점에 넣으면 빈 화면만 보인다.

4. **확장 가장 나중 (V3+)**: 다국어·결제 연동은 국내 소규모 개인 프로젝트 단계에선 불필요하다. 제품이 충분히 검증된 뒤에 판단한다.

## 9. 위험과 가정

| 위험 | 대응 |
|------|------|
| V0.1 검증 실패 (실제 가족 모임에서 안 쓰임) | V0.2 이후 계획을 재검토한다. 피벗 포인트는 첫 번째 실전 모임 이후다. |
| 카카오 알림톡 정책 변경 | V1.0으로 미뤄 두어 초기 개발 리스크를 줄인다. 발생 시 이메일 초대로 대체한다. |
| 개인 프로젝트 공수 과부하 | 한 버전 안에서 기능 수를 3~4개로 제한한다. 초과 시 다음 버전으로 이월한다. |
| 닉네임 기반 지인 매핑 오류 (V2.0) | V0.2의 signed token이 닉네임 일관성을 높이므로 V2.0 전에 반드시 완료한다. |
